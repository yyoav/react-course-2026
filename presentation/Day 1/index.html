What is React?

A JavaScript library for building user interfaces — primarily used for Single Page Applications.

🧩

Component-based

Build UIs from small, reusable pieces.

📝

Declarative

Describe what to show — React handles the DOM.

⚡

Virtual DOM

Updates only what changed — stays fast.

Key distinction: React is a LIBRARY, not a framework (unlike Angular). It handles just the View layer. "Declarative" means you describe the end state and React figures out the DOM changes. Virtual DOM is a lightweight JS representation — diffing algorithm compares old vs new tree.

Single Page Applications (SPA)

⚙️ CSR — Client Side Rendering

React runs in the browser. JS renders the UI. This is what the diagram above shows.

🖥️ SSR — Server Side Rendering

Server renders HTML before sending it. Different flow — covered in Next.js.

Traditional multi-page app: every click = full page reload from server. SPA loads ONE HTML shell, then JS handles all navigation. Mention tradeoffs: initial load can be heavier, SEO requires extra work (hence SSR/Next.js later). CSR is the default for Vite + React. Ask class: "Who has used Gmail or Google Maps?" — those are SPAs.

Why React?

⚡ Build faster

Reusable components — write once, use everywhere.

🧹 Stay clean

Declarative code is easier to read and maintain.

🔄 Reactive UI

State changes update the UI instantly — no manual DOM.

🌍 Ecosystem

Router, DevTools, TypeScript, testing — all ready to use.

React is the most widely used UI library — backed by Meta, used at scale worldwide.

Emphasize the job-market angle: React dominates job listings. Ecosystem is massive — React Router, Redux/Zustand, React Query, testing with Vitest/RTL. Compare briefly: Angular = opinionated full framework, Vue = similar but smaller community. React lets you pick your own tools.

What is Vite?

A fast modern build tool for web apps — replaces the old Create React App.

⚡

Instant start

No waiting for a full bundle before dev server starts

🔄

Fast HMR

Hot Module Replacement — see changes without full reload

📦

Optimised build

Production bundle is minified and tree-shaken automatically

Vite (French for "fast") uses native ES modules in dev — no bundling needed. Only the file you changed gets sent to the browser. In production, it uses Rollup under the hood for optimized output. CRA is deprecated since 2023 — Vite is the recommended replacement. Mention esbuild for dependency pre-bundling (Go-based, extremely fast).

Your First Function Component

  • A function that returns JSX
  • Name must start with a capital letter
  • Reusable — use <Greeting /> anywhere
  • export default makes it importable

// Greeting.tsx
import React from 'react';

function Greeting() {
  return (
    <h1>Hello, React! 👋</h1>
  );
}

export default Greeting;

💡 Components let you break the UI into small, independent, reusable pieces.

Stress the naming convention: PascalCase = component (Greeting), camelCase = regular function. React distinguishes components from HTML elements by the capital letter. The `import React` line is no longer needed in React 17+ with the new JSX transform, but shown here for clarity. Ask: "What happens if we name it `greeting` lowercase?" — React treats it as an HTML tag.

JSX

✍️ You write


// Greeting.tsx
function Greeting() {
  const name = 'React';
  return <h1>Hello, {name}!</h1>;
}

⚙️ Under the hood — React 19 JSX transform


import { jsx as _jsx } from 'react/jsx-runtime';
const name = 'React';
_jsx("h1", { children: ["Hello, ", name, "!"] });
JSX is NOT HTML — it's syntactic sugar that compiles to function calls. This transformation is done by Babel (or SWC in Vite) at build time — it converts JSX into regular JavaScript function calls. The new JSX transform (React 17+) means you don't need `import React` at the top anymore. We'll cover the key JSX syntax rules on the next slide.

JSX Rules

Single root element

Wrap siblings in a <div> or Fragment

// ❌ Two roots
return <h1>Hi</h1> <p>Bye</p>;

// ✅ Fragment wrapper
return <><h1>Hi</h1><p>Bye</p></>;
className & htmlFor

HTML reserved words are renamed

<div className="card">   // ✅
<div class="card">      // ❌

<label htmlFor="email"> // ✅
<label for="email">    // ❌
{} for expressions

Embed any JS expression

<h1>{user.name}</h1>          // ✅
<p>{2 + 2}</p>                 // ✅
<img src={avatarUrl} />       // ✅
Walk through each rule. 1) Single root — JSX compiles to a function call, and a function can only return one value; Fragment avoids extra DOM nodes. 2) All tags must close — unlike HTML, JSX is strict like XML. 3) className/htmlFor — because `class` and `for` are reserved words in JavaScript. 4) Curly braces are the "escape hatch" into JavaScript — any expression works (variables, math, function calls, ternaries) but NOT statements (if/for).

App Component (Concept)

  • App.tsx is the root component
  • Child components nest inside parents
  • Builds UIs from small, reusable pieces
  • Every React app has exactly one root
Component Tree 
App
├── Header
├── Greeting
│   └── Avatar
└── Footer
Think of it like a family tree. App is the grandparent. Each component can have children. This tree structure is how React knows what to render and in what order. Draw parallel to HTML DOM tree — same concept, but with your custom components. Ask: "If Greeting re-renders, does Header also re-render?" Answer: by default no, only the subtree.

App Component (Code)


// App.tsx
import Header from './Header';
import Greeting from './Greeting';
import Footer from './Footer';

function App() {
  return (
    <div>
      <Header />
      <Greeting />
      <Footer />
    </div>
  );
}

export default App;

App is the root — it composes Header, Greeting, and Footer into a single UI tree.

Show this in the actual demo project. Open App.tsx and point out: imports at top, function body returns JSX, self-closing tags for components. The wrapping div is needed because JSX must return one root. Mention Fragment shorthand as alternative. Ask students to create their own component file and import it into App.

Props Overview

📦 Data Down

Props flow one-way — from parent to child only.

🔒 Read-Only

A child never mutates its own props.

♻️ Reusable

Different prop values = different component output.

Props are how parents talk to children. One-way data flow is a core React principle — makes debugging easy because data always flows downhill. If a child needs to communicate back, it calls a callback function passed as a prop (we'll see this later with lifting state up).

Props Example

Parent — App.tsx 

// App.tsx
import Greeting from './Greeting';

function App() {
  return <Greeting name="Dan" />;
}

export default App;
Child — Greeting.tsx 

// Greeting.tsx
type GreetingProps = { name: string };

function Greeting({ name }: GreetingProps) {
  return <h1>Hello, {name}!</h1>;
}

export default Greeting;

The type GreetingProps enforces the shape — TypeScript will error if name is missing or the wrong type.

Walk through both sides: child declares what it expects via a type, parent passes the actual value. Show what happens if you omit `name` or pass a number — TS red squiggles. Destructuring in the function signature is the idiomatic React pattern. Mention: you can also use `interface` instead of `type`.

Events in React

HTML 
<button onclick="handleClick()">
  Click me
</button>
React 

// MyButton.tsx
function MyButton() {
  const handleClick = () => {
    alert('Clicked!');
  };

  return (
    <button onClick={handleClick}>Click me</button>
  );
}

export default MyButton;

React uses camelCase prop names and expects a function reference, not a string.

Key differences from HTML: 1) camelCase (onClick not onclick), 2) Pass a function reference, don't call it, 3) No quotes around the handler. React uses synthetic events that wrap native browser events for cross-browser consistency. The event object is passed automatically as the first argument.

Event Handler Rules

camelCase

Use camelCase prop names


<button onClick={fn}>     // ✅
<button onchange={fn}>    // ❌
Pass a function

Don't call it — pass the reference


<button onClick={handleClick}>    // ✅
<button onClick={handleClick()}>  // ❌
Inline arrow

Use when you need to pass args


<button onClick={() => handleClick('hi')}>  // ✅
Common mistake: writing onClick={handleClick()} — this CALLS the function immediately on render instead of waiting for a click. The inline arrow form is useful when you need to pass arguments. Walk through each rule with the class.

State & useState

🗃️ What is state?

Data owned by a component that can change over time.

const [value, setValue] = useState(initialValue)
value

Current state value

setValue

Setter — triggers re-render

initialValue

Used on first render only

🔄 Re-renders on change

Calling the setter tells React to re-render with the new value.

State is the BIG concept of the day. Props = data from outside, State = data owned internally. When state changes, React re-renders the component automatically. The useState hook returns a tuple: [currentValue, setterFunction]. The initial value is only used on the very first render.

useState in Action


// Counter.tsx
import { useState } from 'react';

function Counter() {
  const [count, setCount] = useState(0);

  return (
    <button onClick={() => setCount(count + 1)}>
      Clicked {count} times
    </button>
  );
}

export default Counter;

🏁 Initial Render

useState(0) returns [0, setCount].

UI: "Clicked 0 times"

Walk through the Counter step by step. useState(0) sets initial count to 0. The button's onClick fires setCount with count + 1. Show this live in the demo project — students should follow along.

useState in Action


// Counter.tsx
import { useState } from 'react';

function Counter() {
  const [count, setCount] = useState(0);

  return (
    <button onClick={() => setCount(count + 1)}>
      Clicked {count} times
    </button>
  );
}

export default Counter;

🖱️ Button Clicked

setCount(count + 1) fires.

React queues 0 → 1 & re-renders.

When the button is clicked, setCount is called with count + 1. React doesn't update immediately — it queues the update and schedules a re-render. Note: this direct form works fine for simple cases. We'll see the functional updater form next, which is safer when updates depend on previous state.

useState in Action


// Counter.tsx
import { useState } from 'react';

function Counter() {
  const [count, setCount] = useState(0);

  return (
    <button onClick={() => setCount(count + 1)}>
      Clicked {count} times
    </button>
  );
}

export default Counter;

🔄 Re-render

count is now 1.

UI: "Clicked 1 times" ✓

On re-render, React calls Counter() again. This time useState returns [1, setCount]. The button text updates. This cycle repeats every click. Key insight: the component function runs again from top to bottom on every re-render.

Functional Updates


// ❌ Direct — can read stale value
setCount(count + 1);

// ✅ Functional — always reads latest
setCount(prev => prev + 1);

Use the functional form whenever the next value depends on the previous one.

Why it matters
  • React batches state updates
  • Direct form may read a stale closure value
  • Functional form receives the actual latest state
  • Always safe for counters & accumulators
This trips up many beginners. If you call setCount(count + 1) twice in the same event handler, you get count+1 both times (not count+2) because count is a stale closure. The functional form always receives the actual current value. Rule of thumb: if new state depends on old state, use the functional form.

Rules of Hooks (1/2)

❌ Wrong 
if (isLoggedIn) {
  const [x, setX] = useState(0);
}
for (const id of ids) {
  useEffect(() => fetch(id));
}
✅ Correct 
const [x, setX] = useState(0);
useEffect(() => { /* ... */ });
if (isLoggedIn) { /* use x */ }
📏 Top level only

Never call hooks inside loops, conditions, or nested functions.

React relies on call ORDER to track which hook is which. If you wrap hook #2 in an if-statement that's false, hook #3 shifts to index #2 — state is corrupted.

Rules of Hooks (2/2)

❌ Wrong 
function calcTotal(items) {
  const [sum, setSum] = useState(0);
}
✅ Correct 
function Panel() {
  const [open, setOpen] = useState(true);
  return <div>{open && '...'}</div>;
}
function useToggle(init = false) {
  const [on, setOn] = useState(init);
  return [on, () => setOn(p => !p)];
}
⚛️ React functions only

Only call hooks from function components or custom hooks (use prefix).

🔌 eslint-plugin-react-hooks catches both rules automatically.
Hooks are attached to React's internal fiber tree. A plain JS function has no fiber — useState has nowhere to store state. The `use` prefix is a convention AND enforced by the linter.

State with Objects


const [user, setUser] = useState({ name: 'Dan', age: 30 });

// ✅ spread — creates a new object
setUser(prev => ({ ...prev, age: 31 }));

// ❌ mutate — React won't re-render
user.age = 31;
Why spread?
  • React uses === to detect changes
  • Mutating the same object keeps the same reference
  • Spread creates a new object — React sees the change
  • Only the updated property changes; the rest is copied

Never mutate state directly — always return a new reference so React detects the change.

This is the #1 state bug beginners hit. React uses reference equality (===) to detect changes. If you mutate the same object, the reference doesn't change and React skips the re-render. Spread operator creates a shallow copy — that's a new reference. Show in the console: `const a = {x:1}; const b = a; a === b` is true, but `{...a} === a` is false. That's why spread works.

State with Arrays


const [items, setItems] = useState(['a', 'b', 'c']);

// ✅ add
setItems(prev => [...prev, 'd']);

// ✅ remove
setItems(prev => prev.filter(i => i !== 'b'));

// ✅ update one item
setItems(prev =>
  prev.map(i => i === 'a' ? 'z' : i)
);
✅ Immutable methods

map, filter, spread, slice, concat

❌ Mutating methods

push, pop, splice, sort (in-place), reverse

Same rule as objects — always return a new array so React sees the update.

For arrays: use map/filter/spread, never push/splice directly. The immutable vs mutating table is a handy cheat sheet — the left column returns a new array, the right column modifies in place. If you need sort, do [...items].sort() to copy first. Ask class: "What does filter return?" — a new array with items that pass the test.

Lifting State Up

When two siblings need the same state, move it to their common parent and pass it down as props.

Example 

// Parent.tsx
function Parent() {
  const [name, setName] = useState('');
  return (
    <NameInput value={name} onChange={setName} />
    <NameDisplay name={name} />
  );
}

State lives in Parent — NameInput writes, NameDisplay reads.

Lifting state up is how siblings share data. Find the closest common ancestor, put useState there, pass value down as props and setter as a callback. This pattern appears everywhere in React apps. Preview: when prop-drilling gets deep, we'll use Context (Day 3).
